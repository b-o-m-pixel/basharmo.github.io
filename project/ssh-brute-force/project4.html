01
Attack Simulation
Hydra SSH Brute Force — Kali Linux
hydra · rockyou.txt wordlist · target port 22
Hydra was launched from Kali Linux on the local network targeting SSH (port 22) on the Ubuntu host running Wazuh. The rockyou.txt wordlist simulates a real credential stuffing attack. Hydra fires multiple username/password combinations in rapid succession — exactly the pattern Wazuh's composite brute force rules are designed to detect. This screenshot captures the attack actively running, with attempts being fired at the target.
02
Host-Level Evidence
auth.log — Failed Password Entries
grep "Failed password" /var/log/auth.log
Before relying on the SIEM, this confirms the attack reached the target at the OS level. /var/log/auth.log is Linux's native authentication log — every failed SSH login is recorded with timestamp, source IP, and attempted username. Grepping for "Failed password" isolates only failed attempts, showing Kali's IP repeating across many lines — the raw signature of a brute force. This is a foundational SOC skill: verifying host-level evidence independently of the SIEM.
03
SIEM Detection
Wazuh SSH Attack Monitor Dashboard
Custom dashboard · 4 visualizations · wazuh-alerts-4.x-* index
Custom SSH Attack Monitor dashboard built from scratch in Wazuh, displaying the attack across four visualizations: a total alert count metric, a timeline bar chart showing the spike during the attack window, a data table of top attacking IPs confirming Kali's address, and a pie chart breaking down which SSH rules fired. Each visualization was created individually against the wazuh-alerts-4.x-* index with rule.groups filtered to sshd — this is active SIEM configuration, not passive monitoring.
04
Alert Investigation
Expanded Alert — Full Event Detail
rule.id · data.srcip · timestamp · MITRE ATT&CK mapping
Drilling into a single alert reveals the full event record: rule.id, rule.description, data.srcip confirming Kali's IP, precise timestamp, agent.name, and MITRE ATT&CK T1110 mapping. This is the core SOC workflow — moving from dashboard overview down to a single event to confirm the threat and gather evidence for escalation. Understanding how to read a full alert record is what separates passive monitoring from actual analysis.
05
Detection Logic
Rule 5760 — sshd: Authentication Failed
Wazuh Management → Rules → 5760
Rule 5760 is the base SSH authentication failure rule — fires on every single failed login attempt. During the Hydra attack, each individual attempt triggers this rule, generating the high-volume alert flood visible in the timeline. This is the foundational signal: one failed login. Understanding this rule explains why alert volume is so high during a brute force and why analysts need composite rules to avoid alert fatigue.
06
Detection Logic
Rule 5758 — Maximum Authentication Attempts Exceeded
Wazuh Management → Rules → 5758
Rule 5758 fires when the SSH daemon reports the per-connection retry limit has been exhausted. This is a higher-confidence signal than 5760 — it means the attacker hit the connection's maximum retry cap, a behavioral pattern distinct from a single wrong password. Seeing 5758 alongside 5760 confirms the attack is automated and persistent, not accidental.
07
Detection Logic
Rule 5763 — Brute Force Attempt Detected
Wazuh Management → Rules → 5763 · MITRE T1110
Rule 5763 is the high-confidence composite rule — it fires when Wazuh observes a threshold of failures from the same source IP within a defined time window. This maps directly to MITRE ATT&CK T1110 (Brute Force) and carries high severity, triggering escalation in a real SOC. Understanding how this rule aggregates lower-level events (5760, 5758) into a single high-confidence verdict is the key insight: this is how SIEM correlation reduces noise into actionable intelligence.
